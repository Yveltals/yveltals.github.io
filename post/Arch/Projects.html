CMU 15445

BUFFER POOL

存储管理器中实现缓冲池。缓冲池负责将物理页面从磁盘中读入内存、或从内存中写回磁盘，使得DBMS可以支持大于内存大小的存储容量。页面读入缓冲池时，从free-list（空位页面）后lru_replacer中（未被访问的可替换页面）寻找空间，淘汰的dirty-page还需写回磁盘，page_table映射page到frame的关系

并行缓冲池的思想是分配多个独立的缓冲池，并将不同的页面ID映射至各自的缓冲池中，从而减少整体缓冲池的锁粒度，以增加并行性。负载均衡：采用轮转方法选取分配物理页面时使用的缓冲池，start_idx++

为什么不用 OS 磁盘管理模块

OS 的磁盘管理模块没有 DBMS 中的领域知识，无法恰当地决定数据移动的时机和数量

  • 将 dirty pages 按正确地顺序写到磁盘
  • 定制化缓存置换策略
  • 根据具体情况预获取数据

LRU-K：最近使用过1次 —> 最近使用过K次，降低了“缓存污染”问题，命中率比LRU高

保存每个页面访问次数、最近K次时间戳、链表中的位置

  • 历史队列：访问小于k次的页面，按FIFO存放，每次访问将页面计数++，满k次放入缓存队列
  • 缓存队列：访问满k次的页面，按照前第K次时间排序，用SET结构存放

优先排除历史队列末结点，否则排除缓存队列末结点（前第k次最早）

EXTENDIBLE HASH INDEX

磁盘备份指该哈希表可写入至磁盘中，在系统重启时可以将其重新读取至内存中使用。可扩展哈希表是动态哈希表的一种类型，其思想在于多个目录项对应一个桶，其特点为桶在充满或清空时可以桶为单位进行桶分裂或合并，并且仅需在特定情况下进行哈希表全表的扩展和收缩，以减小扩展和收缩操作对全表的影响

  • 全局深度：哈希键有很多位，全局深度规定了仅右侧几位有效，因此通过改变哈希表目录项的哈希键位数来控制哈希表的总大小，个目录项
  • 局部深度：每个目录项所指的桶关注的哈希键位数，这样当局部深度小于全局深度时，就实现了多个表项映射到一个桶中，防止桶的数量成倍扩张，而且不需要把所有数据全部rehash一遍，只需rehash原分裂桶

桶的分裂：桶满时，根据局部深度 j 和全局深度 i：

  • i>j时：多个目录项指向的桶满，哈希表不用扩展，仅需桶局部深度++分裂成两个，将对应的几个目录项重新映射，一半指向旧桶一半指分裂桶
  • i==j时：全局深度++，该桶局部深度++，导致目录项翻倍，该桶分裂出一个新桶。对应的两条目录项分别指向两个分裂桶，并且把原桶的记录重新hash到两个分裂桶中；其他的目录项低i-1位相同的指向同一个桶。

桶的合并：满足两桶均空、局部深度相同时，合并桶后局部深度-1，合并后可能缩减全局深度

并发控制

  • getvalue：依次哈希目录、桶页面加读锁

  • insert/remove：桶一定会插入或分裂，但目录项只有在桶满分裂时才修改，故采用乐观锁优化，默认目录项只加读锁，桶加写锁；只有桶满时，释放锁并调用另一分裂插入函数，一开始获取两个写锁，并判断桶是否仍满

提示

不难注意到，在Insert中释放读锁和SplitInsert中释放写锁间存在空隙，其他线程可能在该空隙中被调度，从而改变桶页面或目录页面数据。因此，在这里需要重新在目录页面中获取哈希键所对应的桶页面（可能与Insert中判断已满的页面不是同一页面），并检查对应的桶页面是否已满。如桶页面仍然是满的，则分配新桶和提取原桶页面的元数据。

B+Tree Index

  • TREE_PAGE: 保存在page的data中，总大小4KB，包含size、max_size、parent_page_id、page_id
  • TREE_INTERNAL_PAGE: kv数组*<keyType, page_id>*
  • TREE_LEAF_PAGE: next_page_id与kv数组*<keyType, record_id>*

理论定义

  • B+树分为内部结点和叶子结点，根结点既可以是内部结点，也可以是叶子结点，关键字个数最少1个
  • B+树内部结点不保存数据，只用于索引，所有数据都保存在叶结点中
  • m阶B+树内部结点最多有m-1个关键字、m个子树，叶子结点最多存储m-1个记录
  • 内部结点中的key都按照从小到大的顺序排列，key左子树中的key都大，小于等于右子树中的key
  • 叶子结点双向链表相连，自小而大有序

插入

理论上B+树内部结点是由指针、关键字交替组成的，实际的实现是一个键值对数组，键对应关键字，值对应右侧的指针（实际是page_id），所以M阶的B+树内部结点最多存放M个键值对，第一个键值对键是空的，而叶子结点键值对个数为M-1

  1. 如果是空树，创建一个叶节点作为根。更新root_page_id ，因为需要通过它访问B+树
  2. 从根节点向下查找到键值应该所在的叶节点，在每个内部节点找到最后一个小于等于插入值的键，取它的值即就是下层结点的ID，最终找到叶子结点并插入键值对。规定不支持重复键，所以先扫描一遍叶节点，如果发现键存在则直接返回 false
  3. 如果叶节点插入后超过 M-1 要进行分裂，将原节点的一半内容拷贝到新节点，分裂点的键插入父节点，让父节点该键的值指向新的叶节点
  4. 如果父节点（内部节点）插入后超过了M，同样要递归进行分裂并向上插入，还要调整子节点的父指针指向新的内部节点。如果根节点满了，则要创建一个新的根节点，使得 B+ 树长高一层

删除

下溢：内部结点 (M+1)/2，叶结点 M/2

  1. 如果空树就返回。否则找到键所在叶节点，从叶节点中删除键值，删除后出现下溢出就进行后续处理：
  2. 对于叶结点，如果兄弟节点键值对个数大于 M/2，从兄弟节点借一个键值对（左侧兄弟就借尾，右侧兄弟就借头），把兄弟节点的键上移替换父节点中的键；如果兄弟节点不够借出，将该节点与兄弟节点合并，同时删除父节点中该位置的键值。
  3. 如果删除后父节点（内部节点）下溢出，仍然是借&修改或合并&删除两种方法处理，但规则与叶节点不同：借&修改时，要把兄弟节点的键上移，借来的键值对的键取自父节点的键，值取自兄弟节点的值；合并&删除时，要把父节点的键和兄弟节点的键值一块和该节点合并。
  4. 如果下溢出达到根节点，而且根节点只剩下一个子节点，说明树应该减少一层，将这个子节点设为新的根。

并发

安全节点：插入后不上溢，删除后不下溢，读取时均安全。如果子节点安全，那么对其下面的节点做插入/删除操作引起的树结构变化最多会传递到该层，不会影响上层，所以可以放掉祖先节点的锁允许其它操作访问

基础算法：

  • 对于读操作，子节点获取读锁后就释放父节点读锁，在实现上只需要一个 prev_page 指针解锁父节点

  • 对于插入/删除操作，在子节点获取写锁，如果安全就释放祖先节点的所有写锁，否则继续持有。在实现上，用 page_set和 deleted_page_set记录查找过程中加写锁的页面和删除的页面。

    • 对于插入，page_set保存的写锁就足够了，因为新生成的兄弟结点不会被其它操作访问到
    • 对于删除，还需要访问兄弟节点时上写锁，不在 page_set 中，需要单独释放。此外合并页导致删除的页面记录到 deleted_page_set 中，最后调用内存池接口删除掉

  • 另需一把锁保护根节点，只有根结点安全情况才能解锁，否则堵塞前后 root_page_id_ 可能变化了

改进算法

对于插入删除操作，每次都要获取根节点写锁，成为性能瓶颈，而且大多时候并不会分裂合并。所以 “乐观” 地假设插入/删除不会发生分裂或合并，于是只需沿路径像读取一样获取和释放读锁，然后检查叶节点：如果叶节点安全，那么假设正确，对其更新只需要叶节点的写锁；而如果不安全则假设错误，重新调用基础版算法。

提示

Unlock 和 Unpin 的顺序：先 Unlock 后 Unpin，因为 Unpin 后这个 Page 的 pin count 可能降为 0，buffer_pool_manager 可能会将该 Page 指针的内容改写为另一个 Page 的，导致 Unlock 错误

QUERY EXECUTION

一条 sql 语句，首先经过 parser 生成一棵抽象语法树 AST（采用libpg_query 库），Planner 遍历这棵树，生成初步的查询计划，得到一个Plan Tree，每个Plan结点都代表了一个操作，树中上层的Plan结点每次调用 next() 从下层结点取出一条数据，从而数据自底向上地流动，在根节点输出结果。

优化器

由 Planner 得到初步的查询计划后，对 Plan Tree 进行修改优化。优化器遍历原始 Plan Tree，根据已经定义好的一系列规则，对 Plan 结点进行一系列的修改优化等操作。这种优化器不需要知道数据的具体内容，仅是根据预先定义好的规则修改 Plan 结点。

TopN：对 Plan Tree 进行后续遍历，在遇到父节点 Limit、子节点 Sort 时，则将这两个节点替换为一个 TopN

HashJoin：

  • 把 NestedLoopJoin 替换为 HashJoin
  • 让小表驱动大表，减少表连接次数，调用已提供的api估计table大小，再根据大小来调整 Plan Tree 里 join 的顺序
  • 索引识别，默认IndexJoin 里只会尝试为右表匹配 index，左表则被忽略。因此如果左表有 index，右表没有，且为等值连接，则把左右顺序替换一下，便于之后正确识别索引

谓词下推：需要把 Filter 正确下推至 Join 算子下。自顶向下地改写，提取 predicate 中的所有 comparison，判断表达式的两边是否一个是 column value，一个是 const value，只有这样的 predicate 可以被下推，再将所有的 predicate 重新组合为 logic expression 生成新的 Filter，根据 column value 的 idx 来选择下推的分支

列裁剪：遇到连续的两个投影，合并为 1 个，只取上层投影所需列，其余直接抛弃

火山模型

  • 简单，每个 Operator 可以单独抽象实现、不需要关心其他 Operator 的逻辑
  • 数据以行为单位进行处理，不利于CPU cache ；且每处理一行需要调用多次next() 函数，虚函数开销大

HashJoinExecutor

使用基础哈希算法进行连接操作，其原理为将元组的连接键（即某些属性列的组合）作为哈希表的键，并使用其中一个子计划节点的元组构造哈希表。由于具有相同连接键的元组一定具有相同的哈希键值，因此另一个子计划节点中的元组仅需在该元组映射的桶中寻找可与其连接的元组

用unordered_multimap直接存放等于连接键的元组，再用右子节点的元组作为”探针”寻找与其连接键相同的左子计划节点的元组，可能有多个左子结点对应

AggregationExecutor

用哈希表将所有聚合键（即元组的属性列组合）相同的元组映射在一起，以此统计所有聚合键元组的聚合信息

  • ExtendibleHash

事务层面

由 LockManager使用 2PL 实现了具体的元组级锁，采用严格2PL协议，commit时才一次性释放所有锁，避免级联回滚问题。当一个事务需要读写元组时候，需要根据自身的隔离级别尝试获得元组对应的读锁或写锁，并在适当的时刻将其释放。

隔离级别：

隔离级别问题加锁模式
可重复读幻读GROWING上锁，SHINKING解锁
提交读幻读、不可重复读读锁(立即释放)+写锁
未提交读幻读、不可重复读、脏读仅写锁

数据结构：

  • 每个 transaction 保存了持有的读写锁、隔离级别、修改索引/元组的记录
  • 每个 tuple 对应的RID有一个加锁请求队列，用条件变量堵塞/唤醒请求，加锁时堵塞，解锁时唤醒。

死锁预防：严格2PL有死锁风险，如事务1请求A,B的锁，事务2请求B,A的锁。采用Wound-Wait策略：当优先级高的事务等待优先级低的事务的锁时，将优先级低的事务杀死；当优先级低的事务等待优先级高的事务的锁时，优先级低的事务将阻塞。其思想在于避免互相等待锁。

锁的操作：

  • 加读锁：判断合法（事务abort、2PL冲突、不符合隔离级别），加入请求队列尾部，从前开始遍历，如果前面没有加写锁的事务则获取到锁，否则若事务更晚则kill掉并唤醒队列上等待的事务，若更早的则堵塞

  • 加写锁：判断合法，只要有任何一个锁请求就会造成阻塞，杀死更晚的事务并唤醒队列上等待的事务，遇到更早的堵塞

  • 升级锁：找到对应读锁的请求删除，在第一个未授予锁的请求之前插入写锁请求（优先级应该最高），之后的处理跟加写锁类似，当本请求为队列的第一个时得到锁，否则这样等前边所有锁都释放后就可以获得锁

  • 解锁：唤醒其他事务，仅在可重复读事务级别转为shinking，其他事务不限制，为了兼容性始终growing

并发控制：

为查询计划执行器的顺序扫描、插入、删除计划的NEXT()方法提供元组锁的保护，使得支持并发执行。

  • seq_scan：元组加读锁，未提交读不加锁，提交读读完立即解锁，可重复读在COMMIT或ABORT时释放锁

  • insert：元组插入表时由 tablePage 锁保证并发安全，插入后为了更新索引，需要先获得元组的写锁，保证此期间元组不被修改，之后再更新索引依靠 B+tree 的锁

  • delete：调用 next() 收到子计划节点要删除的元组后（如seq_scan），应当为该元组加写锁，然后标记元组删除，并删除索引，等到事务提交时缓冲池删除。注意在 RR 级别时，子计划节点此时拥有该元组的读锁，因此应该采用升级锁

Bug

SeqScanExecutor用table_itor遍历元组时，每次访问元组前后加解读锁，而每次next结束时itor++指向下一个元组，会自动跳过已经被删除的元组，但仍然出现本线程获得读锁之后访问到一个已经删除的tuple。

原因：itor++后、本线程还未获取itor指向的tuple的读锁前，其他线程这时将itor指向的tuple删除了。所以需要获取读锁后，再使用GetTuple读一次tuple的值，如果false则跳过该已被删除的tuple

CS 144

Router

  • 实现一个路由器，对应网络层的数据平面，根据输入的规则，按照最长前缀匹配规则转发

ARP

  • 实现一个network interface，也就是网卡以及arp协议
  • 网卡中有一个arp表，实现ip地址和mac地址的转换，它通过arp协议自动生成，每个表项都会过期
  • 借由链路层的广播帧，可以将arp分组发送给所有人，从而得到预期的回复

发送帧

  1. 地址缓存机制，主要用于检查目标地址 next_hop 是否已经缓存（新增一个地址缓存映射）
    • 已缓存，将数据包转换成 EthernetFrame ，push 到 frames_out 队列中
    • 未缓存，广播局域网 ARP 消息（其实就是 push 一个 EthernetFrame ，但是 payload 是 ARPMessage 结构序列化的 buffer）
  2. 对相同地址发送 ARP 广播查询，需要间隔 5s
  3. 当前准备发出的请求，如果由于未知地址没有发出，需要先缓存到队列中（新增一个请求队列）

收到帧

  1. 如果 frame 是 ipv4 数据包，解析成 InternetDatagram类型
    • 如果数据包的目标 ip 不是自己的，返回 {}，反之返回解析成功的数据包
  2. 如果 frame 是 arp 数据包
    • 解析成 ARPMessage ，并且保存发送端的 ip 地址和 mac 地址映射（如果已经保存则重置过期时间）
    • 如果 ARP 请求的是自身的 ip，发送一个 ARP Reply
    • 检查可以发送的 请求队列

TCPConnection

TCP有限状态机：三次握手，正常通信，四次挥手

优雅断开连接：双方都发送完才能关闭，还要考虑是否等待2MSL，因为此时没有明确的客户/服务端

  1. 接收流已经组装完成并且终止（接收到了FIN）
  2. 发送流已经将所有内容发送给对方（包括FIN），并且全部得到了确认
  3. 先发送FIN的一方要确认对方是否收到了ACK，所以满足前两个条件后仍要TIME_WAIT，而另一方不用。所以设置 linger 标识来区分在满足条件1、2的时候是否可以连接立即结束，否则要在收到最后一个报文段后等待规定时间之后才结束。
    • 区分方式为：默认 linger 标识为true，表示需要TIME_WAIT
    • 如果输入流先结束（收到了FIN 且 Reassembler为空），输出流后结束（应用层调用ByteStream的end_input()方法结束输出且ByteStream为空），此标识被设为false，无需TIME_WAIT

强制断开连接：在未达到上述三个条件的时候强行关闭，发送RST报文。发送方直接退出连接，接收方将本地的stream设置为error，active变为false。

  • 将收到的包传给StreamReassembler
  • 计算出正确的ackno和window_size，以告知发送方实现流量控制(flow-control)

TCP Sender

  • TCP可靠数据传输机制的实现
    • 定时器(单一的重传定时器+队列保存发送但未确认的数据)
    • 序号和确认(确认是被捎带在数据报文段中的)
  • TCP连接管理:SYN和FIN
  • 流量控制：根据接收方的窗口大小调整发送量（空/探测报文也要确认并返回窗口，窗口调整后要立即填充）

计时器超时后：

  • 重传已发送未确认的最早的segments。

  • 如果window size不为0，进行拥塞控制：

    • 跟踪连续重传的次数，来决定连接是否需要中止
    • 将RTO的值翻倍，称作指数避退，它减少了糟糕网络上的重传，以避免进一步破坏网络状况。

  • 重启计时器，并在RTO时间后再次重复以上的过程

收到ackno后：

  • 发送窗口向右移动到ackno所在的位置（将发送窗口中被完全确认的segments删除）
  • 说明网络拥塞状态有所缓解，RTO重置为初始值、重传次数归0，重启计时器
  • 如果sender没有任何未确认的segments，关闭计时器

Reassembler

  • 负责把收到的包（有重叠、无冲突）根据序号重组，按序写入ByteStream
  • 抛弃过去的包和超出容量的包
  • 储存容量容许范围内，但失序的包

ByteStream

  • FIFO模型
  • 有限容量
  • 支持一端写入一端读出

模拟网络传输逻辑

首先，项目根路径中的 tun.sh 会使用 ip tuntap 技术创建虚拟 Tun/Tap 网络设备。这类接口仅能工作在内核中。不同于普通的网络接口，没有物理硬件。这样做的目的应该是为了模拟真实网络环境下的网络环境。

当 Tun/Tap 网络设备建立好后，tcp_ipv4.cc 中会建立一个 TCPOverIPv4OverTunFdAdapter。TunFd指的是连接进 Tun 设备上的 socket，而TCPOverIPv4OverTunFdAdapter是一个 IP 层面的封装接口。当调用 adapter 向其写入 TCP 报文段时，它会自动 wrap 成 IP 段并传输进网络设备中；读取也是亦然，会自动解除 IP 段并返回其内部封装的 TCP报文段。

接下来，无论对于 Server 还是 Client，在三次握手之后，都会建立一个新的线程，来专门执行 LossyTCPOverIPv4SpongeSocket 中的 eventloop。而子线程会另起一个 eventloop 以及另外开辟两个缓冲区，用于存放用户写入的数据与即将输出至屏幕的数据。当用户通过 stdin 输入数据时， eventloop 中所注册的 poll 事件被检测到，则数据将会被写入进本地输入缓冲区中；同样，也可以将收到的数据从ByteStream中读出，允许接收窗口增大。当 TCPOverIPv4OverTunFdAdapter 可写时，它会将本地输入缓冲区中的数据全部写入至 TCPOverIPv4OverTunFdAdapter 并最终传输至远程。

而 webget 与真实服务器通信的原理，也是通过将 IP 报文写入 tun 虚拟网络设备，将其注入进 OS 协议栈中，模拟实际的发包情况。

TCP报文传输方式

  • TCP-in-UDP-in-IP：用户只需提供 TCP 包，之后可以使用 Linux 提供的接口，让内核来负责构造 UDP 报头、IP报头以及以太网报头，并将构造出的数据包发送至下一个层。因为这一切都是内核完成的任务，因此内核可以确保每个套接字都具有本地地址与端口、远程地址与端口的唯一组合，同时能保证不同进程之前的隔离。
  • TCP-in-IP：Lab4机制 通常，TCP数据包是直接放进 IP 包作为其 payload，这也因此被称为 TCP/IP。但用户层如果想直接操作构造 IP 报文的话，需要使用到 Linux 提供的 TUN 虚拟网络设备来作为中转。当用户将 IP 报文发送给 TUN 设备后，剩余的以太网报头构造、发送以太网帧等等的操作均会由内核自动进行，无需用户干预。
  • TCP-in-IP-in-Ethernet：Lab5所使用的机制 上面两种方式仍然依赖Linux内核来实现的协议栈操作。每次用户向TUN设备写入IP数据报时，Linux 内核都必须构造以太网帧、找出下一跳MAC地址，如果 Linux 无法得知该映射关系，则将会发出广播探测请求以查找到下一跳的地址。针对这部分功能，实现了网络接口和ARP协议的机制，在发送前先封装ARP分组并广播，对应主机返回单播响应告知MAC地址，这样网络接口就能把待发送的 IP 报文转换成以太网帧了，之后把帧发送给 TAP 虚拟网络设备，剩下的发送操作将会由它来代为完成。

难点/Bug

  1. TCP的状态机涉及到12种状态间的转换，需要判断和区分各种状态，同时排除非法报文，要求很强的鲁棒性，需要很多的细节逻辑来控制
  2. 序列号和流下标的转换，syn和fin也占 seqno 但不占 streamIndex，测试用例中syn、fin和payload一同发送的情况下要做好判断
  3. 收到ack后更新window没自动填充；收到没数据的数据包keep-alive要回复空报文；streamID和abs_seqID；关闭连接的原理

接收string数据的处理流程：TCPConnection -> TCPReceiver -> Reassembler -> ByteStream。需要先把乱序的报文临时存在重组器中，对string进行切分、排序、合并（大量的拷贝），若和接收缓冲区相邻再拷贝进缓冲区中。期间涉及 string 大量的copy与substr，故优化的核心思想就是避免复制string本身，而是复制智能指针，避免真正切分，而是偏移量标记

1. Buffer封装string

  • 智能指针实现 string 所有权转移：make_shared<string>(move(str)) ，所有接口的参数传递都用智能指针Buffer
  • 被读取或修建时，用 _starting_offset 和 _ending_offset 标识有效string的两端，因此substr的操作用Buffer类的remove_prefix，remove_suffix避免了拷贝
  • 需要向外暴露内容时，用string_view()

2. 更改ByteStream缓冲区结构TODO

  • 使用deque<Buffer>而不是 deque<char>
  • 直接移动 starting_offset 位置即可实现流的读出，避免了substr
  • 向string中append string_view的对象会比string快，可以把Buffer转型成string_view再append到返回值中

3. 优化Reassembler算法

  • 原先算法中，Block{size_t index, string data}，二分查找+合并
    • 先从set (底层红黑树) 中利用二分搜索找到可以合并的节点（不能合并就直接加进去）
    • 找到的话，计算合并之后的结果 ， 删除旧节点，加入结果
    • 逆序效率很低
  • 优化后，Block{size_t index, Buffer data}，二分查找+不真正合并
    • 能合并的话，不更改已储存的包，而是把这个新包修剪一下，去掉它的重复的部分
    • 字符串在别的地方存着，Buffer用两个offset，把substr操作从线性时间优化为常数时间

4. 改造接口：右值引用+共享指针

  • 三次调用本来都是string& 但这样不可避免的会调用复制，如传入重组器时Buffer转string，重组器写入bytestream时，Buffer转string
  • 所以把这些函数的接口改成Buffer结合智能指针

Webserver

项目性能以及瓶颈

瓶颈在于网络设计模式的选择：单Reactor多线程模式。一个Reactor对象承担了所有事件的监听和响应，当遇到突发的高并发时，往往不能及时处理新连接。

针对瓶颈如何改进

采用多Reactor多线程模型。主Reactor只负责监听连接还有建立连接，并且将建立好的连接通过生产者消费者模型传递给子线程里的子Reactor负责监听对应连接上的事件。这样主线程就可以在遇到瞬间并发时也能够及时处理新连接的建立。

Webbench是什么

父进程fork若干个子进程，每个子进程在用户要求时间内对目标web循环发出访问请求。父子进程通过管道进行通信，子进程通过写端向父进程传递在若干次请求访问完毕后记录到的总信息，父进程通过读端读取。子进程在时间到后结束，父进程在所有子进程退出后统计并给用户显示最后的测试结果，然后退出。

事件处理模式

Reactor模式

Reactor 模式是非阻塞同步网络模式，感知的是就绪可读写事件。主线程只监听文件描述符上是否有事件发生，有的话就立即将该事件通知工作线程，由工作线程负责完成读写数据、接受新的连接、以及处理客户请求。这个过程是同步的，只有先读取完数据后才能处理数据。

使用同步 I/O 模型实现的 Reactor 模式的工作流程，以 epoll_wait 为例：

  1. 主线程往 epoll 内核事件表中注册 socket 上的读就绪事件。
  2. 主线程调用 epoll_wait 等待 socket 上有数据可读。
  3. 当 socket 上有数据可读时， epoll_wait 通知主线程。主线程则将 socket 可读事件放入请求队列。
  4. 睡眠在请求队列上的某个工作线程被唤醒，它从 socket 读取数据，并处理客户请求，然后往 epoll 内核事件表中注册该 socket 上的写就绪事件。
  5. 主线程调用 epoll_wait 等待 socket 可写。
  6. 当 socket 可写时，epoll_wait 通知主线程。主线程将 socket 可写事件放入请求队列。
  7. 睡眠在请求队列上的某个工作进程被唤醒，它往 socket 上写入服务器处理客户请求的结果。
  8. 工作线程从请求队列中取出事件后，将根据事件类型 event 来决定如何处理它，没必要区分所谓的 “读工作线程” 和 “写工作线程”。

Proactor模式

Proactor 是异步网络模式，感知的是已完成的读写事件。所有 I/O 操作都交给主线程和内核来处理，工作线程仅仅负责业务逻辑。在发起异步读写请求时，只需要传入数据缓冲区的地址（用来存放结果数据）等信息，内核完成读写工作后才通知应用进程直接处理数据。

优缺点

  • Reactor实现相对简单，对于耗时短的处理场景处理高效；可以依次同步的执行事件而不需要加锁；但处理耗时长的操作会造成阻塞，影响到后续事件处理；
  • Proactor性能更高，允许多个任务并发的执行从而提高吞吐量，可执行耗时长的任务（各个任务间互不影响）；但实现逻辑复杂。

HTTP解析怎么实现的

  1. 采用reactor处理模式，主线程只负责监听IO，获取IO请求后把请求对象放入请求队列，工作线程负责数据读取以及逻辑处理。

  2. 工作线程通过有限状态机解析HTTP请求

    将程序中的不同状态进行整合，来保证不论状态发生的顺序如何，最后都能转移到需要的状态上。

    项目中状态机主要用于对客户端请求的处理，三种状态包括处理HTTP请求行、请求头、请求体。通过在状态机里重复这三个部分之间的解析跳转，最终可以完成一个完整请求报文的解析。一方面可以避免多种状态同时发生造成混乱，另一方面保证了正确的状态转移，比 if 判断更可靠。

  3. 正则表达式

    HTTP1.1请求报文是ASCII码的固定文本形式，通过正则表达式去解析可以简化我们的任务量。

HTTP应答怎么将网页资源发送出去

采用集中写的形式。利用Linux函数 writev()，我们可以根据解析结果制作响应行和响应头并且存放在缓冲区，然后将响应体（请求的资源文件）通过内存映射mmap()映射到内存中去，最后通过集中写的形式将一个完整的HTTP应答发送出去

说一说线程池

这个项目实现一个固定线程数量的线程池用来管理多线程。使用到了生产者消费者模型，所有线程共享一个生产队列，工作线程之间对任务队列的竞争采用条件变量和互斥锁结合使用。

一个工作线程先加互斥锁，当任务队列中任务数量为0时候，阻塞在条件变量，当任务数量大于0时候，用条件变量通知阻塞在条件变量下的线程，这些线程来继续竞争获取任务。为了可以同时处理多个连接，需要在让线程的处理函数不断while循环。

所有线程是分离状态，在线程池销毁的时候设置关闭标志位，唤醒所有线程使其退出。

对任务队列中任务的调度采用先来先服务算法

如何更好的使用线程池

线程池本质上是生产者和消费者模型，包括三要素：

  1. 往线程池队列中投递任务的生产者；

  2. 任务队列；

  3. 从任务队列取出任务执行的工作线程（消费者）。

线程池的线程数量

  • 对于CPU密集的任务，线程池的线程数量最好为CPU处理器个数+1。
  • 对于IO密集的任务，一般要多于CPU的核数，线程间竞争的不是CPU资源而是IO，IO的处理比较慢，多于核数的线程将为CPU争取更多的任务，不至于在处理IO的时候造成CPU空闲。
  • 如果任务执行时间长，工作线程很快就很被任务占完，导致后续不能及时处理，应增加工作线程数量。
  • 如果任务执行时间短，那么工作线程数量不用太多，否则线程上下文切换浪费时间。

定时器的作用

高并发的服务器通常会面临大量的连接建立，但是实际活跃的连接并不多，这样会造成服务器端的资源浪费。

需要模拟TCP层面的keep-alive套接字属性。默认的超时时间过长，通过在应用层设置定时器来模拟该选项，可以及时关闭失效的连接来避免资源的浪费。

怎么用小根堆实现定时器

小根堆是组织定时器的一个数据结构，每次取队头都是最短的超时时间，以最短的超时时间来作为定时，确保每次定时时间到达都有事件发生。

小根堆支持对定时器的增删改，通过哈希表将文件描述符映射到指定的定时器中，实现了 O(1) 时间复杂度的查找，但是堆调整则需要 logn 的时间复杂度。

日志系统

日志系统通过单例模式、双缓冲区和异步线程，实现日志文件的异步写入。通过使用宏定义简化日志系统的使用。

双缓冲区是什么

工作线程在执行过程中将要输出的日志内容输出到一个 buffer 缓冲区中。同步则直接将 buffer 缓冲区的内容写入到对应文件中去；异步则将其转移到生产者消费者队列中去，同时唤醒异步线程将其取出写入到对应的日志文件中去。

双缓冲区是为了缓解内存读写速度和IO读写速度不匹配的问题，同步线程不需要，因为当前进程会被IO阻塞，不存在读写速度不匹配的问题。但是当开启异步线程时，IO读写速度和内存读写速度不匹配，容易导致数据丢失，所以开启异步线程时需要使用双缓冲区。
