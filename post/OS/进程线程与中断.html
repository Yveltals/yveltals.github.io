进程

进程属性

进程属性保存在进程控制块 (PCB)的结构体中，内核通过管理PCB来调度进程

1.标识符: pid进程、ppid父进程、uid用户

2.页表指针: 虚拟内存和物理内存的映射关系

3.进程组(process group): 一个进程(领导进程 pgid) 和其各种附加进程

4.会话(session): 进程组的集合，其中每个进程组称为一个工作(job)。其中创建会话的进程为领导进程，对应会话id即 sid


进程控制

fork()

创建子进程，子进程获得父进程的数据空间、堆栈、页表，对应虚拟内存和物理内存相同，通过读时共享、写时复制共享数据

exec()

用可执行文件取代该进程中的程序，即用新程序中数据替换了进程中代码段、数据段以及堆栈中数据

exit()

exit(0) == return 0，调用exit()函数后，进程非马上消失，而是留下一个成为僵尸进程的数据结构，记录退出状态等信息供父进程收集，若不被收集将一直处于僵尸态

wait()与waitpid()

wait(int* status) ==> wait(NULL)

waitpid(pid_t pid, int* status, int option) ==>waitpid(p,NULL,0)


进程间通信

管道通信

管道通信方式的中间介质是文件，通常称这种文件为管道文件。写进程通过写端（发送端）往管道文件中写入信息；读进程通过读端（接收端）从管道文件中读取信息。

无名管道PIPE：非永久性，PIPE()创建，当它访问的进程全部终止时被撤消，不能用于不同族系的进程之间;
有名管道FIFO：永久，MKNOD()创建，可以弥补PIPE的不足。

通过系统调用WRITE()和READ()进行读写操作，CLOSE()将管道文件关闭。


消息队列

消息队列是消息的链表，存放在内核中并由消息队列标识符表示。消息队列提供了一个从一个进程向另一个进程发送数据块的方法，每个数据块都可以被认为是有一个类型，接受者接受的数据块可以有不同的类型。

我们可以通过发送消息来避免命名管道的同步和阻塞问题。但是消息队列与命名管道一样，每个数据块都有一个最大长度的限制。Linux用宏MSGMAX和MSGMNB来限制一条消息的最大长度和一个队列的最大长度。

  • 生命周期随内核，消息队列会一直存在，需要我们显示的调用接口删除或使用命令删除
  • 消息队列可以双向通信
  • 克服了管道只能承载无格式字节流的缺点

内核为每个IPC对象维护了一个数据结构struct ipc_perm，用于标识消息队列，让进程知道当前操作的是哪个消息队列。每一个msqid_ds表示一个消息队列，并通过msqid_ds.msg_first、msg_last维护一个先进先出的msg链表队列，当发送一个消息到该消息队列时，把发送的消息构造成一个msg的结构对象，并添加到msqid_ds.msg_first、msg_last维护的链表队列。在内核中的表示如下：


共享内存

在Linux中，每个进程都有属于自己的进程控制块（PCB）和地址空间（Addr Space），并且都有一个与之对应的页表，负责将进程的虚拟地址与物理地址进行映射，通过内存管理单元（MMU）进行管理。两个不同的虚拟地址通过页表映射到物理空间的同一区域，它们所指向的这块区域即共享内存。

共享内存的通信原理示意图：

对于上图的理解是：当两个进程通过页表将虚拟地址映射到物理地址时，在物理地址中有一块共同的内存区，即共享内存，这块内存可以被两个进程同时看到。这样当一个进程进行写操作，另一个进程读操作就可以实现进程间通信。但是，我们要确保一个进程在写的时候不能被读，因此我们使用信号量来实现同步与互斥。

线程

线程简介

每个线程（thread）是CPU使⽤的⼀个基本单元；它包括线程ID、程序计数器、寄存器组和堆栈。它与同⼀进程的其他线程共享代码段、数据段和其他操作系统资源，如打开⽂件和信号。如果⼀个进程具有多个控制线程，那么它能同时执⾏多个任务。 从这个⻆度来看，线程就是⼀个进程的执⾏进度。

下图可以看到，⼀个线程有⾃⼰的寄存器和堆栈，这是程序执⾏最基本的要素。即线程是程序执⾏的基本的单位，或者说， 线程本身就是⼀个进程的执⾏部分 。所谓CPU调度的本质就是选择合适的线程放到⼀个CPU核⼼上来执⾏。因此，在每个CPU核⼼上执⾏的是⼀个个线程，⽽不是进程本身（不区分线程的进程也可以看做是只有⼀个线程的进程）。

同⼀个进程的线程之间⼜使⽤相同的 IO资源 、 信号 、 内存资源 等，这些资源都是以操作系统以进程为单位分配的。于是，进程被称为是资源分配的最⼩单位


使⽤线程的好处

  • 响应性 。如果浏览器是⼀个只有⼀个线程的进程。如果此时⽹络发⽣了延迟，那整个浏览器的所有操作都将停滞
  • 资源共享 。进程只能通过共享内存和消息传递之类的技术（消息队列、管道等）共享资源。⽽同⼀个进程的线程之间天然可以共享同⼀个虚拟地址空间的代码、数据、全局变量等等，这相⽐于进程来说，更加⾼效和限制更⼩。⽐如，进程之间数据的共享要通过内核来进⾏，⽤户地址空间的代码访问⼀次内核空间（系统调⽤）的代价是很⼤的；⽽且共享的数据⼤⼩还要受到内核地址空间⼤⼩的限制，⾮常不⽅便。
  • 经济 。进程创建所需的内存和资源分配⾮常昂贵。由于线程能够共享它们所属进程的资源，所以创建和切换线程更加经济。进程创建和管理⽐线程花费更多时间。
  • 可伸缩性 。对于多处理器体系结构，多线程的优点更⼤，因为线程可在多处理核上并⾏运⾏。不管有多少可⽤CPU，单线程进程只能运⾏在⼀个CPU上。

并发的意思是多个任务快速交替执⾏，让⼈以为它们是在同时执⾏。
并⾏的意思是多个任务在同时执⾏，这种情况出现在多核系统上。


线程实现

使⽤线程的两个主要动机

  1. 充分利⽤多个CPU核⼼：⼀个应⽤程序通常作为具有多个控制线程的⼀个进程来实现，在⼀个CPU的多个核⼼上并⾏执⾏多个CPU密集型的任务
  2. 减少切换进程的昂贵开销：⼀个应⽤程序可能需要同时执⾏多个相似任务

针对这两点开发出了两种线程： 内核线程 和 ⽤户线程 。两者分类的区别主要在于其线程调度器是在内核中还是在内核外 。

内核线程：由于CPU调度时由内核控制的，所以只有内核线程才能被真正分配到不同的CPU核⼼上去执⾏，从⽽达到充分利⽤CPU核⼼的能⼒。

⽤户线程：调度器在内核外，也就是说，⽤户线程的创建和调度都是由上层的应⽤程序使⽤ 线程库 的⽅式⾃⼰来完成的，这种线程库⼀般是对操作系统底层系统调⽤的封装。


多对⼀模型

在这种线程模型中，操作系统中仅有 ⽤户线程 。也就是说，操作系统本身只提供对进程的⽀持，⽽没有对线程的⽀持。⼀个进程中可以有多个线程，但这些线程都是由应⽤程序⾃⼰使⽤⾃⼰的线程库来调度的。内核看不⻅这些线程，内核仅仅知道存在这么⼀个进程，⾄于进程内部发⽣了什么，⼀概不知。

所以，当⽤户线程只能用进程的身份来访问内核，内核在调度的时候也只能使⽤去调度进程。因此，如果在访问内核的过程中阻塞了，那整个进程的其他线程此时就⽆法访问内核了，⾮常不⽅便。


一对一模型

充分利⽤内核对 内核线程 的调度能⼒，每个进程都把⾃⼰的每个线程都映射到⼀个内核线程上，从⽽使得内核对内核线程的调度也就相当于对每个进程⾃身线程的调度。

这种线程模型实现起来简单，Windows和Linux都实现了这种线程模型。但缺点是，内核要为每个线程创建⼀个内核级线程与对应，⽽操作系统对本系统能容纳的线程数是有限制的。


多对多模型

将上述两种线程模型融合起来，但实现起来很复杂。很多商业操作系统都实现了
这种线程模型，如Solaris。


轻量级进程

Linux是⼀个很“简单”的系统，在理解相关概念的时候尽量不要混⼊诸如“⽤户线程”、“内核级线程”之类的概念，就重点理解⼀个 轻量级进程 就够了，实际上，这就是关于Linux线程的全部。
Linux中没有对进程和线程进⾏区分，⽽是统⼀使⽤ task_struct 来表示。线程和进程都是task ，都能被 同等调度 。

  • 当⼀个 task 独享资源（IO资源、信号、内存资源等）时，就是重量级进程
  • 当它与其他进程共享这些资源时，就被称为轻量级进程 ，也就是所谓的线程

从字⾯上就可以看出，这个轻量级进程就是缺失了⼀些资源的进程。

线程创建函数 pthread_create 具体实现的时候，会调⽤ clone 系统调⽤，这个系统调⽤的⽤途就是内核⽤来创建进程的。因此，在 pthread_create 调⽤ clone 的时候，可以通过设置标志位，创建出⼀个与⽗进程共享资源的⼦进程，这个⼦进程很显然就是⼀个 轻量级进程 （因为它与其他进程共享资源），从⽽完成创建⼀个新的线程的任务。
值得⼀提的是， clone 内部最终会调⽤ do_fork 函数，创建进程具体的逻辑就在这个函数中实现；⽽ fork 函数最终也会去调⽤ do_fork 。创建进程和线程时间线在这⾥收束！


内中断

内中断的产生

  1. 除法错误:0
  2. 单步执行:1
  3. 执行 into 指令:4
  4. 执行 int 指令,格式为int n ,n为字节型立即数,是中断类型码

中断过程

由中断类型码找到中断向量,并自动用它设置 CS和IP，随后执行中断处理程序

  1. 从中断信息取得中断类型码N
  2. pushf //标志寄存器入栈(中断过程中值改变)
  3. TF=0,IF=0 //设置标志寄存器的TF、IF位
  4. push CS push IP
  5. (IP)=(N*4),(CS)=(N*4+2)

中断向量表

中断向量表是中断处理程序入口地址的列表，在内存0000:0000到0000:03FF的1024个单元中存放，一个表项存放一个中断向量,占两个字,高地址存放段地址,低地址存放偏移地址

中断处理程序

中断处理程序必须一直存储在内存某段空间之中，而中断向量必须存储在对应的中断向量表表项中

中断处理程序的编写方法和子程序的比较相似,下面是常规的步骤:

  • 保存用到的寄存器;
  • 处理中断;
  • 恢复用到的寄存器;
  • 用 iret 指令返回。

外中断

什么是外中断

外设随时都可能发⽣需要CPU及时处理的事件，这种中断来⾃于CPU的外部，当CPU外部有需要处理的
事情发⽣的时候，⽐如说，外设的输⼊到达，相关芯⽚将向CPU发出相应的中断信息。CPU在执⾏完当
前指令后，可以检测到发送过来的中断信息，引发中断过程，处理外设输⼊。

内外中断的区别

简单来说，外中断和内中断的区别就是，外中断产⽣于外设，⽽内中断是由程序执⾏时产⽣的。

可屏蔽中断和不可屏蔽中断

在PC系统中，外中断源⼀共有以下两类。

  • 可屏蔽中断

当CPU检测到可屏蔽中断信息时，如果IF=1，则CPU在执⾏完当前指令后响应中断，引发中断过程：如果IF=0，则不响应可屏蔽中断。

可屏蔽中断所引发的中断过程，除在第1步的实现上有所不同外，基本上和内中断的中断过程相同。因为可屏蔽中断信息来⾃于CPU外部，中断类型码是通过数据总线送⼊CPU的；⽽内中断的中断类型码是在CPU内部产⽣的。

故中断过程中将IF置为0的原因：在进⼊中断处理程序后，禁⽌其他的可屏蔽中断。当然，如果在中断处理程序中需要处理可屏蔽中断，可以⽤指令将IF置1。指令：sti ，设置IF=1；cli ，设置IF=0

  • 不可屏蔽中断

当CPU检测到不可屏蔽中断信息时，则在执⾏完当前指令后，⽴即响应，引发中断过程。

⼏乎所有由外设引发的外中断，都是可屏蔽中断。
